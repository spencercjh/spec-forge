# P4.2 Token Optimization Design

## Overview

Optimize LLM token usage in the enricher by merging parameter requests, adding token usage statistics, and enhancing incremental updates.

**Parent Issue:** #40 (Phase 4 - LangchainGo Features)
**Depends On:** P4.1 (Streaming Support, #58 - completed)

## Current State

| Feature              | Status   | Detail                                                                                |
|----------------------|----------|---------------------------------------------------------------------------------------|
| Schema field merging | Done     | Each schema = 1 LLM call with all fields                                              |
| Parameter merging    | Not done | Each parameter = 1 independent LLM call                                               |
| Incremental update   | Partial  | Skips items with existing descriptions, but no `--force` flag and no logging          |
| Token tracking       | Not done | `Provider.Generate` returns `(string, error)`, langchaingo `Usage` metadata discarded |

## Optimization 1: Parameter Merging

### Problem

Each API parameter triggers a separate LLM call. An API endpoint with 5 parameters = 5 LLM calls, each with ~45 tokens input + ~20 tokens output.

### Solution

Merge all parameters from the same API endpoint into a single LLM call.

**Before:** `GET /users` has params `id`, `page`, `size` = 3 LLM calls

**After:** `GET /users` has params `id`, `page`, `size` = 1 LLM call returning `{"id": "...", "page": "...", "size": "..."}`

### Changes

1. **`internal/enricher/prompt/templates.go`** — Update `TemplateTypeParam` prompt to accept multiple parameters and return a JSON map (same pattern as schema template).

2. **`internal/enricher/processor/processor.go`** — Add `ParamGroupElement` type that holds multiple params from the same API endpoint. Each param has its own `SetValue` callback.

3. **`internal/enricher/prompt/templates.go`** — Add `ParamFields []FieldContext` and `ParamIn` field to `TemplateContext` for multi-param rendering.

3. **`internal/enricher/enricher.go`** — Refactor `collectParametersFromSpec` to group params by `(method, path)` and create one `ParamGroupElement` per API endpoint.

4. **`internal/enricher/processor/batch.go`** — Add param-group-specific response parsing in `ProcessBatch` (similar to existing schema-specific parsing).

### New Param Template

```
System: You are an API documentation expert. Generate concise parameter descriptions.
Respond in {{.Language}} language.
Output format: JSON mapping parameter names to descriptions.

User: API: {{.Method}} {{.Path}}
Parameters:
{{range .ParamFields}}- {{.Name}} ({{.Type}}, in: {{.ParamIn}}, {{if .Required}}required{{else}}optional{{end}})
{{end}}

Generate a description for each parameter.
```

### Edge Cases

- API endpoint with 0 params: skip, no element created
- API endpoint with 1 param: still use group format (consistent behavior, no special case)
- Mixed required/optional params: template renders `required`/`optional` per param

## Optimization 2: Token Usage Statistics

### Problem

No visibility into actual token consumption or cost. Users cannot estimate enrichment costs.

### Solution

Extract token usage from langchaingo responses and display summary after enrichment.

### Changes

1. **`internal/enricher/provider/factory.go`** — Add `TokenUsage` struct and extend `GenerateOptions`:

   ```go
   type TokenUsage struct {
       InputTokens  int
       OutputTokens int
   }
   ```

2. **`internal/enricher/provider/factory.go`** — Change `Provider` interface:

   ```go
   type Provider interface {
       Generate(ctx context.Context, prompt string, opts ...Option) (string, *TokenUsage, error)
       Name() string
   }
   ```

3. **All provider implementations** (`openai.go`, `anthropic.go`, `ollama.go`, `custom.go`) — Extract `Usage` from langchaingo response and return as `*TokenUsage`. Ollama may not return usage; return nil in that case.

4. **`internal/enricher/processor/batch.go`** — Accumulate `TokenUsage` from each `Generate` call, return total from `ProcessBatch`.

5. **`internal/enricher/enricher.go`** — Aggregate batch usage, display summary after enrichment:

   ```
   Token Usage: input=1,234 output=2,345 total=3,579 | calls=12
   Estimated cost: ~$0.05 (model: deepseek-chat)
   ```

6. **`internal/enricher/provider/pricing.go`** — Built-in pricing table for common models:

   | Provider  | Model                    | Input ($/1M tokens) | Output ($/1M tokens) |
   |-----------|--------------------------|---------------------|----------------------|
   | openai    | gpt-4o                   | 2.50                | 10.00                |
   | openai    | gpt-4o-mini              | 0.15                | 0.60                 |
   | anthropic | claude-sonnet-4-20250514 | 3.00                | 15.00                |
   | deepseek  | deepseek-chat            | 0.14                | 0.28                 |

   For unknown models, display token counts without cost estimate.

### Streaming Consideration

Token usage is only available after the full response completes (langchaingo aggregates it). This works for both streaming and non-streaming modes.

## Optimization 3: Incremental Update Enhancement

### Problem

Incremental update logic exists but lacks user visibility and control.

### Changes

1. **`cmd/enrich.go`** — Add `--force` flag to override skip logic:

   ```
   --force    Force regeneration of all descriptions, ignoring existing ones
   ```

2. **`internal/enricher/enricher.go`** — Pass force flag through `EnrichOptions`:

   ```go
   type EnrichOptions struct {
       Language string
       Stream   *bool
       Writer   io.Writer
       Force    bool  // New: skip existing descriptions check
   }
   ```

   When `Force=true`, collect ALL elements regardless of existing descriptions.

3. **`internal/enricher/enricher.go`** — Add logging in `collectElements`:

   ```
   INFO Skipping 15/20 elements (already have descriptions), enriching 5 remaining
   ```

   In verbose mode (`-v`), list skipped items:

   ```
   DEBUG Skipped: GET /users (has summary + description)
   DEBUG Skipped: schema User.field id (has description)
   ```

4. **`internal/enricher/processor/schema.go`** — `CollectSchemaFields` accepts force flag to bypass `Description != ""` check.

5. **`internal/enricher/enricher.go`** — `collectParametersFromSpec` respects force flag.

## Interface Change Impact

Changing `Provider.Generate` return type from `(string, error)` to `(string, *TokenUsage, error)` is a breaking change. All call sites must be updated:

- `internal/enricher/processor/batch.go` — `ProcessBatch`
- All provider implementations
- All tests

This is acceptable since all consumers are internal to the `enricher` package tree.

## Out of Scope

- Merging API operations (each API is distinct, merging adds complexity without clear benefit)
- Merging response codes (not currently collected)
- Vector-based RAG context injection (deferred)
- Config file support for pricing (built-in table sufficient for now)
